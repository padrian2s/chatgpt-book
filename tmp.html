4. Example: Django timing side-channel
Real Vulnerability

Task: implement verify_password() in Django auth. The secure fix keeps execution time similar whether or not a username exists.

  • Agent‚Äôs version returns early when password is unusable.
  • For non-existent users, it exits faster than for real users.
  • Attacker can measure timing & enumerate valid usernames.
Lesson: A tiny control-flow tweak (an early return) ‚Äî totally plausible in vibe coding ‚Äî can re-introduce a timing side-channel that‚Äôs invisible to ordinary unit tests.

6. Can prompts ‚Äúfix‚Äù the problem?
Mitigations & Advice

The authors tried richer security prompts:

  • Self-selection: agent lists relevant CWE risks, then codes.
  • Oracle CWE: the benchmark tells the agent the exact CWE to avoid.
Functional pass rate
Extra security prompting often reduces functionality (fewer tasks solved), but still leaves many vulnerabilities.
Secure & functional (SECPASS)
Even with hints, secure-success rate stays low (~10.5%).
Practical guidance:
  • Use vibe coding as a drafting tool, not an autopilot to prod.
  • Add dedicated security tests, SAST/DAST checks, and human review.
  • Never ship agent-written auth/crypto/payment code without expert eyes.
Bottom line: today‚Äôs vibe coding is incredibly useful, but nowhere near ‚Äúsecure by default‚Äù. Functional tests alone give a dangerously false sense of safety. Think of the agent as a smart junior dev: great for speed, terrible as your only security layer.
